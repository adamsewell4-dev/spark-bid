# Proposal Generation Rules
# Applies to: src/proposals/**

## Voice & Tone
- Write in first-person plural ("Digital Spark Studios will deliver...")
- Professional, confident, government-contractor register
- Avoid marketing hyperbole — GSA reviewers respond to specifics, not superlatives
- Always define acronyms on first use (FAR, NAICS, SOW, PWS, etc.)

## Required Proposal Sections (in order)
1. Executive Summary — 1–2 paragraphs, restate the requirement and DSS's fit
2. Technical Approach — How DSS will execute the specific work requested
3. Management Plan — Joshua Hieber (Executive Director) as POC, Adam Sewell (Executive Producer) as backup
4. Past Performance — Pull from the 4 verified references in the database
5. Pricing Narrative — Structure, not numbers (numbers come from the pricing sheet)
6. Compliance Matrix — One row per stated RFP requirement, with DSS response

## Past Performance Matching Logic
- Training/educational video RFPs → lead with Qworky + Atlas Copco references
- Brand/campaign RFPs → lead with Carnegie Mellon University reference
- Commercial/advertising RFPs → lead with SleepMe reference
- Multi-service RFPs → use all four references

## Quality Control Language (reuse verbatim)
Pre-Production: client consultation, script/storyboard approval, equipment readiness checks
Production: on-set supervisors, regular footage review, lighting/audio/framing checks
Post-Production: multi-cycle editing review, color grading + audio mastering, client approval before delivery
Responsible personnel: Joshua Hieber (Executive Director) + Adam Sewell (Executive Producer)

## Hard Rules
- Never invent past performance — only use verified references from the database
- Never claim certifications DSS does not hold
- Always flag requirements DSS cannot meet — do not fabricate coverage
- Subcontractor note when relevant: "Subcontractors will perform less than 50% of total work in compliance with FAR 52.219-14"
- FAA Part 107 drone certification applies only when aerial work is in scope
